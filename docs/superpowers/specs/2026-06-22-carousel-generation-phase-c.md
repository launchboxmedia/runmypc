# Carousel Generation — Phase C Spec

**Date:** 2026-06-22
**Status:** Approved (autonomous build authorized by user)
**Branch:** `carousel-phase-c`

## Context

Final phase of the per-customer carousel feature. Depends on:
- **Phase A** (merged): `lib/designSystem/` — `resolveDesignSystem`, `STYLE_LIBRARY`, `derivePalette`, `persistJobStyle`. Schema columns on `profiles`/`jobs` (`style_id`, `primary_color`, `split_image_cover`).
- **Phase B** (live): `hyperframes-render` service `render_mode:"static"` → POST HTML, returns a single PNG (4:5, 1080×1350, deviceScaleFactor 2) with the 7 style font families bundled as base64 `@font-face` (zero network egress). The service owns typography.

Phase C **replaces** the existing Remotion carousel (`content-generation.ts` Step 4) with a design-system-driven pipeline that produces on-brand slide PNGs and stores them under the **same `job_outputs` schema** so the dashboard (`ContentSection.tsx`) and `download-carousel` route keep working unchanged.

## Locked decisions

1. **Cover visual = pixels only, HTML = all text.** The image model (Atlas/GPT-Image-2) generates a *visual-only* background/subject (no text). The static render composites headline/handle/badges as real design-system HTML text on top. Reliable fonts across all 5 styles.
2. **Cover visual source:** if the job has a selected approved image asset (`usable_in` static/both), use it as the cover visual. Otherwise generate **2 variants** and vision-score to pick the winner. Asset path skips scoring.
3. **Per-slide HTML generated by Haiku** from the resolved style descriptor + design tokens (mirrors the existing `hyperframes.ts` agent-composition pattern). Strict contract: only bundled font families, palette colors, 1080×1350 root, safe margins, no external resources.
4. **Visual quality gate (legibility + render correctness):** Haiku vision reads each rendered PNG — text legible, not clipped/overflowing, fonts rendered (not generic fallback), contrast OK. Fail → regenerate that slide's HTML with the failure note, re-render. **Max 2 retries**, then keep the best attempt.
5. **Dynamic slide count, info-driven (~5–7).** **Slide 1 = Cover (Hook)** always. **Last slide = single CTA** always. Middle = the IG post's insight lines, each one beat. **No double CTA** — exactly one CTA, the final slide.
6. **Copy source:** derive from the existing Instagram `social_post` output (`hook` / `body` / `cta`) — no new copy LLM call. Split `body` into beats; clamp middle beats so total lands 5–7.
7. **No Remotion.** Remove `renderCarousel` and its call site.
8. **Network-egress constraint:** the render lambda has no guaranteed egress. The cover visual is fetched server-side and **embedded as a base64 data-URI** inside the cover HTML (never an external `<img src>`). Same rule for any embedded image.

## Architecture — `lib/carousel/`

Small single-purpose units. Pure logic is unit-tested; I/O units are injectable + live-verified.

```
lib/carousel/
  types.ts            # SlideBeat, SlidePlan, CarouselSlideResult, GenerateCarouselResult
  buildSlidePlan.ts   # PURE: IG post {hook,body,cta} -> ordered SlidePlan (cover + middle + cta), clamp 5-7
  renderClient.ts     # renderStaticPng(html,w,h) -> PNG Buffer (POST render_mode:"static")
  coverVisual.ts      # resolveCoverVisual(...) -> { dataUri } | null  (selected asset OR 2 generated + scored)
  scoreVisual.ts      # pickBestVisual(images, ctx) -> index  (Haiku vision; injectable)
  slideHtml.ts        # generateSlideHtml(...) -> HTML string  (Haiku per style; injectable)
  qualityGate.ts      # checkSlide(pngBuffer, slide) -> { pass, issues }  (Haiku vision; injectable)
  generateCarousel.ts # orchestrator: resolve -> plan -> cover visual -> per slide: html -> render -> gate(retry<=2)
```

### `types.ts`

```ts
import type { ResolvedDesignSystem } from '@/lib/designSystem/resolveDesignSystem'

export type SlideBeat = 'hook' | 'value' | 'cta'   // cover uses 'hook'; middle beats 'value'; final 'cta'

export type SlidePlan = {
  index: number          // 0-based; 0 = cover
  beat: SlideBeat
  isCover: boolean
  text: string           // the line of copy for this slide
}

export type CarouselSlideResult = {
  index: number
  beat: SlideBeat
  png: Buffer
}

export type GenerateCarouselResult = {
  resolved: ResolvedDesignSystem
  slides: CarouselSlideResult[]   // ordered; cover first, cta last
}
```

### `buildSlidePlan.ts` — PURE (TDD)

```ts
export function buildSlidePlan(post: { hook: string; body: string; cta: string }): SlidePlan[]
```

Rules:
- Slide 0 = cover, `beat:'hook'`, `text = post.hook` (fallback: first body line).
- Middle = `post.body` split into non-empty lines, each stripped of bullet glyphs (`•-✅→` + leading numbering). Each becomes a `beat:'value'` slide.
- Final = `beat:'cta'`, `text = post.cta` (fallback `'Follow for more'`).
- **Clamp so total ∈ [5,7]:** middle count clamped to `[3,5]` (total = middle + cover + cta ∈ [5,7]). If fewer than 3 usable body lines, pad is NOT invented — minimum becomes whatever exists (floor of 1 middle → total 3) rather than fabricate copy. (Clamp upper at 5 middle; drop extras.)
- No duplicate CTA: the cta line is never also emitted as a middle beat.

### `renderClient.ts`

```ts
export async function renderStaticPng(html: string, width = 1080, height = 1350): Promise<Buffer>
```

POST `{ render_mode:'static', html, width, height }` to `HYPERFRAMES_RENDER_URL`. Response `{ url }` (PNG on blob). Fetch the URL → return Buffer. Throws on non-200 or missing url. If `HYPERFRAMES_RENDER_URL` unset → throw (caller's Step 4 catches and marks the step skipped/failed).

### `coverVisual.ts`

```ts
export type CoverVisualDeps = {
  generateImage: (p: { prompt: string }) => Promise<{ url: string }>
  pickBestVisual: (images: Buffer[], ctx: { topic: string; styleId: StyleId }) => Promise<number>
}
export async function resolveCoverVisual(input: {
  resolved: ResolvedDesignSystem
  topic: string
  audience?: string | null
  selectedAssetUrl?: string | null   // signed URL of a selected approved static asset, if any
}, deps?: Partial<CoverVisualDeps>): Promise<{ dataUri: string } | null>
```

- If `selectedAssetUrl` present: fetch bytes → return `{ dataUri }` (base64). No generation, no scoring.
- Else: build a **visual-only** prompt from `STYLE_LIBRARY[styleId].hook_technique` + `layout_descriptor` + topic + audience, explicitly instructing **no text/letters/words** in the image, transparent or style-appropriate background, subject per the style's hook device. Generate **2** variants (parallel). Fetch both buffers. `pickBestVisual` → winner index. Return winner `{ dataUri }`.
- Any failure (no key, gen error) → return `null`. Cover then renders text-only over the palette background (still valid).

### `scoreVisual.ts`

```ts
export async function pickBestVisual(images: Buffer[], ctx: { topic: string; styleId: StyleId }): Promise<number>
```

One Haiku vision call with both images (base64). Asks which better fits the topic + style as a **text-free** carousel cover background (relevance, composition, no embedded text, clean subject). Returns 0 or 1. Never throws → defaults to 0.

### `slideHtml.ts`

```ts
export type SlideHtmlDeps = { generate: (prompt: string) => Promise<string> }
export async function generateSlideHtml(input: {
  resolved: ResolvedDesignSystem
  slide: SlidePlan
  handle?: string
  coverVisualDataUri?: string | null   // only for the cover
  retryNote?: string                    // appended on a quality-gate retry
}, deps?: SlideHtmlDeps): Promise<string>
```

Haiku writes a complete standalone HTML document. System constraints baked into the prompt:
- Root element exactly `1080×1350` px, `overflow:hidden`, background = `resolved.background`.
- Use ONLY these `font-family` names (bundled in the render service): the style's `typography.display_font` + `body_font` (exact names from `STYLE_LIBRARY`). No `@import`, no Google Fonts links, no external CSS/JS/images.
- Colors: text/primary = `resolved.primary_color`, accent = `resolved.accent`, bg = `resolved.background`.
- Apply the style's `layout_descriptor` + `typography.treatment` + `hook_technique` for visual character.
- Safe margins ≥ 64px; text must fit (no clipping); large legible type.
- **Cover only:** if `coverVisualDataUri` provided, include `<img id="cover-visual" src="__COVER_VISUAL__">` positioned per the style/`split_image_cover` (split = visual fills one half, solid `background` block the other half holds the headline). The literal token `__COVER_VISUAL__` is string-replaced post-generation with the data-URI (kept out of the prompt for size).
- Handle pill uses `@${handle}` when provided.
- `retryNote` (if set) is appended: "Previous render failed QA: <issues>. Fix specifically: enlarge/relayout so all text is fully visible and legible."

Post-process: replace `__COVER_VISUAL__` with the data-URI (or strip the img if none). Strip markdown fences if Haiku wraps the HTML.

### `qualityGate.ts`

```ts
export type QualityResult = { pass: boolean; issues: string }
export async function checkSlide(png: Buffer, slide: SlidePlan, deps?: { vision: ... }): Promise<QualityResult>
```

One Haiku vision call on the PNG (base64). Checks: all text fully visible (not cut off/overflowing), legible at thumbnail and full size, fonts look intentional (not a broken/blank fallback), text–background contrast adequate, the slide's `text` is present. Returns `{ pass, issues }`. Never throws → on error returns `{ pass:true, issues:'' }` (fail-open so a flaky vision call never blocks delivery; the render itself is deterministic).

### `generateCarousel.ts` — orchestrator

```ts
export async function generateCarousel(input: {
  job: { id: string; topic: string; target_audience?: string|null; outcome?: string|null;
         style_id?: string|null; primary_color?: string|null; split_image_cover?: boolean|null }
  profile: { style_id?: string|null; primary_color?: string|null; split_image_cover?: boolean|null;
             instagram_handle?: string|null } | null
  igPost: { hook: string; body: string; cta: string }
  selectedAssetUrl?: string | null
}): Promise<GenerateCarouselResult>
```

Flow:
1. `resolved = await resolveDesignSystem({ job, profile })`.
2. `plan = buildSlidePlan(igPost)`.
3. `cover = await resolveCoverVisual({ resolved, topic, audience, selectedAssetUrl })`.
4. For each slide (sequential to bound concurrency/cost):
   - `html = generateSlideHtml({ resolved, slide, handle, coverVisualDataUri: slide.isCover ? cover?.dataUri : null })`.
   - `png = renderStaticPng(html)`.
   - `qa = checkSlide(png, slide)`; if `!qa.pass` and attempts < 2 → regen html with `retryNote: qa.issues`, re-render. Keep last attempt after cap.
5. Return `{ resolved, slides }`.

Never throws out of a single slide failure mid-batch beyond the retry cap — a slide that still fails QA is kept (best effort). A hard render/network error propagates so Step 4 marks the step failed.

## Error handling

- Missing IG post → step `skipped` (unchanged).
- `resolveDesignSystem`/`derivePalette` never throw (Phase A guarantees).
- `resolveCoverVisual` failure → `null` → text-only cover (valid).
- `pickBestVisual`/`checkSlide` failure → safe defaults (index 0 / pass) — never block.
- `renderStaticPng` hard failure → propagates → Step 4 `failed` (matches current behavior).
- Haiku HTML wrapped in fences or with stray prose → post-processor extracts the `<html>…</html>` (or `<!doctype…`).
